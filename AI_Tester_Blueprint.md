## Foundation Model

Covers the foundational concepts and rules for QA professionals working with AI systems.

### 🧠 What Changed After 2020 – The Era of Scale

Before 2020, AI models were like smart calculators – good at one thing. After 2020, AI models became like super interns – they can write, read, code, explain, summarize, and even reason.

#### Real-World Example
- Old AI: A spam filter only tells you "spam or not"
- New AI: Can read the email, explain why it is spam, rewrite it professionally, and suggest a reply

AI after 2020 is defined by scale.
- Models like ChatGPT, Gemini, Midjourney are extremely large
- They consume massive compute, electricity, and data
- Public internet data is becoming a limiting factor

#### Two Major Consequences of Scale

1. **Capability Explosion**
   - Models can now perform many tasks well
   - New applications became possible (coding, writing, planning, vision)

2. **Model-as-a-Service Emergence**
   - Training large models is expensive and centralized
   - A few organizations build models
   - Everyone else consumes them via APIs

➡️ Result: AI application demand increased while entry barriers dropped

---

### 🤖 What Is AI Engineering?

AI Engineering = building applications on top of existing foundation models.
⚠️ Using powerful pre-built AI brains to solve real problems == AI Engineering.
You are not training the brain from scratch.
You are:
- Giving it instructions
- Giving it context
- Checking its answers
- Connecting it to real systems

#### Why AI Engineering Is Growing Fast
- Powerful models already exist
- APIs make usage simple
- Minimal infrastructure required
- Even non-engineers can build AI apps

AI Engineering is now one of the fastest-growing engineering disciplines.

**Traditional ML Engineer:**
- Collects 1M records
- Labels data
- Trains a model
- Tunes hyperparameters

**AI Engineer:**
- Chooses GPT / Claude / Gemini
- Writes prompts
- Adds documents via RAG
- Evaluates responses
- Deploys fast

This is why AI Engineering feels faster and more creative.

## From Language Models to Large Language Models (LLMs)

Paper - arxiv.org/pdf/1706.03762 (1%)

### What Is a Language Model?
A language model is like autocomplete on steroids.
`cat will sit on _`

Query -> Tokenised - Chunked -> small token -> Weighted Match -> you will be given a predictive answer.

A language model:
- Learns statistical patterns of language.
- Predicts the next token given context.
- "My favorite color is ___" -> blue
- LLMs do the same thing — but with:
  - Entire internet-level knowledge
  - Reasoning patterns
  - Multiple languages

**Prompt:** "Write test cases for login page"
**LLM Completion:**
- Valid login
- Invalid password
- Locked user
- Rate limiting

This is just prediction — not understanding — but it looks intelligent.

---

### Tokens (Very Important)
Token = basic unit (word, sub-word, or character)
Models don't see words, they see tokens.
`cat will sit on the _`

#### Why does the LLM not use Word as a token?
- chatgping -> chat + gpt + ing
- platform.openai.com/tokenizer
- lunary.ai/deepseek-tokenizer

#### Why tokens?
- Capture meaning better than characters.
- Smaller vocabulary than words.
- Handle unknown/new words.

www.reddit.com/r/LocalLLaMA/comments/1fi5uwz/no_model_x_cannot_count_the_number_of_letters_r/

### Two Types of Language Models

#### 1. Masked Language Models (BERT-style)
Masked Model (BERT) = Fill in the blanks in an exam paper
Bidirectional Encoder Representations from Transformers
- Predict missing tokens anywhere
- Uses left + right context
- Best for understanding tasks

Examples: Sentiment analysis, Classification, Code understanding
https://www.geeksforgeeks.org/nlp/masked-language-models/
"Pramod Is `<Mask>` looking?" -> smart, good, handsome....

#### 2. Autoregressive Language Models (GPT-style)
Autoregressive Model (GPT) = Write an essay one word at a time
- Predict next token only
- Generate text sequentially
- Best for generation

Examples: ChatGPT, Code generation, Writing, summarization
➡️ We mainly focus on autoregressive models

**BERT-style (Masked LM)**
"Playwright is a MASK automation tool."
Output: "Playwright is a browser automation tool."

**GPT-style (Autoregressive LM) - Generative Pre-trained Transformer**
Input: "Playwright is a"
Output: "Playwright is a modern end-to-end testing framework that supports Chromium, Firefox, and WebKit."

---

### When to Use Which (Practical Guidance)

**Use BERT-style models when you need:**
- Text classification
- Named Entity Recognition (NER)
- Semantic search / embeddings
- Similarity matching
- Intent detection

**Use GPT-style models when you need:**
- Chatbots or assistants
- Test case generation
- Bug report writing
- Code generation
- Reasoning over multiple steps
- Natural language explanations

## Hallucinations (LLMs "Making Things Up")

**Definition:** A hallucination occurs when an AI model produces confident but factually incorrect or completely fabricated output, even though the information does not exist in the system, logs, API, or requirements.
The key danger: the answer sounds correct.

### QA Example 1 – Bug Report Hallucination
**Prompt to AI:** "Analyze this screenshot and generate a Jira bug report."
**AI Output (Hallucinated):**
- Error Code: AUTH_403_INVALID_TOKEN
- Root Cause: Token expiry after 15 minutes
- Affected API: /auth/refresh
**Reality:**
- Screenshot only shows "Something went wrong"
- No error code visible
- No token logic involved

Why this is a hallucination: AI invented technical details, no evidence in input, over-confident explanation.

### QA Example 2 – Test Case Hallucination
**Prompt:** "Generate test cases for VWO dashboard."
**AI Output:**
- Test case: Verify Dark Mode toggle
- Test case: Validate Export to CSV
**Reality:**
- No Dark Mode
- No Export feature in product
- Failure Type: Feature hallucination

### QA Red Flags for Hallucinations:
- Mentions features not in PRD
- Invents error codes
- Claims logs show X (without logs)
- Gives exact numbers without evidence
- Sounds too certain without sources

## Anti-Hallucination Rules

**ROLE:** You are a QA assistant operating under strict verification rules.

**SCOPE OF KNOWLEDGE:** You may ONLY use information explicitly provided in PRD, API documentation, Logs, Screenshots, Test data, User input.

**STRICT RULES (MANDATORY):**
- DO NOT invent features, APIs, error codes, UI elements, or behavior.
- DO NOT assume default or "typical" system behavior.
- If information is missing or unclear, respond with: "Insufficient information to determine."
- Every assertion must be traceable to provided input.
- If a detail is inferred, label it explicitly as: "Inference (low confidence)".
- Output must be deterministic and repeatable.

**PROCESS YOU MUST FOLLOW:**
- Step 1: Extract verifiable facts from the input.
- Step 2: List unknown or missing information.
- Step 3: Generate output ONLY from Step 1 facts.
- Step 4: Perform a self-check for hallucinations or contradictions.

**OUTPUT FORMAT (STRICT):**
```
Verified Facts:
Missing / Unknown Information:
Generated Output:
Self-Validation Check:
```

## Inconsistencies (AI Contradicting Itself)

An inconsistency occurs when the AI gives different or conflicting answers for the same requirement, test case, or scenario.

- **QA Example 1 – Inconsistent Test Cases:** Same scenario → different expected behavior.
- **QA Example 2 – Inconsistent Requirement Interpretation:** Session expires exactly at 30 minutes vs 25–30 minutes.

### Prevent Hallucinations (Best Practices)
1. **Ground the Model:** Use RAG (PRD, API specs, logs, screenshots). Force Evidence-Based Output.
2. **Reduce Inconsistencies:** Use Lower Temperature (0–0.2), Structured Output (JSON), Single Source of Truth.

- Whatever biases, errors, or gaps exist in data appear in the model.
- Cutoff date: The model doesn't know events after training ended.
- Transformer Architecture: Attention Mechanism (Query, Key, Value).
